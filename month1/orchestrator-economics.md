# WEAVE Orchestrator Economics

WEAVE is designed so that the runtime can create applications on top of existing Livepeer workloads while routing more value toward orchestrators and keeping prices lower for users.

## The core idea

A normal AI application company sits between the user and the compute layer.

That middle layer usually has to pay for:
- a full product team
- engineering salaries
- marketing and sales
- executive overhead
- investor return expectations

WEAVE replaces that middle layer with an agent-operated runtime.

The runtime still has real operating costs, but they are much smaller:
- agent compute
- scoped contributor work
- infrastructure and hosting
- reinvestment into the next application

Atumera operates WEAVE as steward and does not extract profit from the runtime.

## The middleman tax

In a normal AI application company, the user is not only paying for compute.

They are also paying for the middle layer:
- product organization
- engineering overhead
- marketing and sales
- executive structure
- investor return expectations

That structure takes value away from both sides that matter most here:
- the user pays more than the underlying workload requires
- the compute layer receives less than the user actually spent

WEAVE is designed to remove that middleman tax as much as possible.

## Why this is better for orchestrators

Because the runtime is not built as a profit-taking middleman, the same underlying inference can support:
- a lower price to the user
- more value routed toward orchestrator benefit

Current working model:
- about `10%` is retained for runtime operations and reinvestment
- about `90%` is routed toward orchestrator benefit

The important point is not only the percentages. It is the structure.

The structure is better because the runtime is not trying to capture a large company margin on top of the workload.

## What the orchestrator gets

The orchestrator should get:
- real users paying for real applications built on Livepeer workloads
- more value from the same underlying inference path
- a runtime that handles much of the product, outreach, and iteration burden
- clearer public reporting about what the system is doing

The point is not just to route work to orchestrators. The point is to route more of the economic value of that work toward them.

## Why this is better for users

Users get access to real applications built on Livepeer workloads without paying for the full overhead of a traditional app company.

That means the runtime can offer:
- cheaper applications
- clearer pricing logic
- a system that is easier to explain publicly

## Why this matters for the program

This program should be legible to the people who approved it.

That means the public docs should make the model easy to inspect:
- what WEAVE is building
- how the runtime works at a high level
- why the economics are better for orchestrators
- why the pricing can be better for users
- what will be reported publicly as the runtime goes live
